# Productive OpenClaw — 总体架构文档

> **项目**: Productive OpenClaw v2.0
> **架构**: SRIA递归推理引擎 + SMIM多Agent基础设施 + Platform Core TypeScript后端 + OpenClaw补丁层
> **版本**: 2.0.0 | **日期**: 2026-06-03

---

## 1. 系统架构总览

Productive OpenClaw 是一套 OpenClaw 扩展架构与优化补丁系统，整合了 Python 递归推理引擎和 TypeScript 后端服务引擎，以 OpenClaw 为基础运行时，提供多Provider统一适配、Agent集群并行执行、记忆引擎和协调器协议等增强能力。

### 1.2 技术栈

| 层级 | 技术 | 用途 |
|------|------|------|
| **Patches** | Shell / TypeScript | OpenClaw运行时补丁 |
| **Platform Core** | Node.js + Express + TypeScript | 后端服务引擎 |
| **SRIA-SMIM** | Python 3.9+ + SQLite | 核心推理引擎 |
| **OpenClaw** | Node.js | 基础Agent运行时 |
| **数据库** | SQLite (主) + 向量扩展 | 状态/记忆/监控持久化 |
| **消息协议** | 3DACP (自定义) | 三维坐标消息路由 |
| **通信协议** | REST / SSE / WebSocket / Internal | 服务间通信 |

## 2. 模块划分与交互

### 2.1 模块关系图

```
                    ┌──────────────┐
                    │   用户请求    │
                    └──────┬───────┘
                           │
           ┌───────────────┼───────────────┐
           │               │               │
           ▼               ▼               ▼
    ┌────────────┐  ┌────────────┐  ┌────────────┐
    │   Patches   │  │Platform Core│  │SRIA-SMIM   │

           │               │               │
           │    ┌─────────┘    ┌──────────┘
           │    │              │
           ▼    ▼              ▼
```

### 2.2 交互模式

| 交互方向 | 方式 | 说明 |
|----------|------|------|
| Patches → OpenClaw | 文件补丁 | 运行时补丁自动应用到 OpenClaw 源码 |
| Platform Core → OpenClaw | HTTP API | 作为独立服务，OpenClaw 通过 HTTP 调用 |
| Platform Core → Ollama | HTTP API | 桥接到本地 Ollama 服务 |
| Platform Core → 云Provider | HTTP API | 统一适配器调用各Provider API |
| SRIA-SMIM → Platform Core | 子进程/subprocess | Python引擎作为独立进程被TS层调用 |
| SRIA-SMIM → Ollama | HTTP API | 直接调用本地模型 |
| 3DACP → 所有模块 | 消息总线 | 三维坐标消息路由到目标服务 |

### 2.3 与 OpenClaw 的关系

> Productive OpenClaw 并非 OpenClaw 的替代品，而是其**上层扩展系统**。OpenClaw 提供底层的 Agent 调度、工具调用、安全沙箱等运行时能力；Productive OpenClaw 在其之上构建了：
> - 多Provider统一适配层
> - Python递归推理引擎
> - 记忆引擎与防遗忘系统
> - 蜂群协调框架
> - 运行时优化补丁

## 4. 配置矩阵

| RAM | 4-8 GB |

## 5. Agent集群模式

### 5.1 模式 1: 并行分析

```
用户问题 → 分发器 → [Agent A, Agent B, Agent C]
                    ↓
              结果聚合器 → 综合答案
```

**适用场景**: 多维度分析、头脑风暴、方案对比

### 5.2 模式 2: 流水线

```
用户问题 → Agent A (分析) → Agent B (推理) → Agent C (验证)
                                                  ↓
                                            最终答案
```

**适用场景**: 复杂任务分解、质量管控、学术研究

### 5.3 模式 3: 专家委员会

```
用户问题 → [数学专家, 物理专家, 代码专家]
                    ↓
              协调器 → 综合答案（标注各专家贡献）
```

**适用场景**: 跨学科问题、需要多领域知识的复杂任务

## 6. 数学物理特化模式

### 6.1 增强提示模板

**数学模式**:
- 所有数学推导必须逐步展示
- 使用 LaTeX 格式表示公式
- 明确标注假设和前提条件
- 验证每一步的正确性
- 如果存在多种解法，比较它们的优劣

**物理模式**:
- 所有物理量必须标注单位
- 区分理论预测和实验验证
- 明确标注近似条件和适用范围
- 引用相关物理定律和方程
- 讨论结果的物理意义

### 6.2 特化路由规则

| 输入特征 | 启用模式 | 路由目标 |
|----------|----------|---------|
| 检测到数学关键词 | 数学模式 | 深度模型（14B+） |
| 检测到物理关键词 | 物理模式 | 深度模型（14B+） |
| 数学+物理混合 | 双模型协作 | 快速模型预处理 + 深度模型推理 |

## 8. 技术决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 推理引擎语言 | Python | 丰富的ML生态，快速原型 |
| 后端语言 | TypeScript/Node | 与OpenClaw同构，快速迭代 |
| 数据库 | SQLite + 向量扩展 | 零配置、单文件、适合本地应用 |
| 消息协议 | 3DACP (自定义) | 三维坐标路由，灵活可扩展 |
| Agent框架 | 自研 (OpenClaw兼容) | 灵活可控，深度集成记忆和优化 |
| 本地模型 | Ollama | 成熟稳定，模型生态丰富 |
